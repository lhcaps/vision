# Patch 17.2 — Real Upload Ingestion Buffer Fix

**Date**: 2026-05-03
**Status**: COMPLETED
**Type**: Runtime bug fix
**Phase**: Phase 17 — Real Media Processing

---

## Mission

Fix the root cause preventing real uploads from completing end-to-end through the thumbnail pipeline. Specifically: `MediaService.upload()` passed `file.buffer!` to `buildMediaIngestionPlan()`, but `diskStorage` produces `file.path` (no `file.buffer`). This caused the MIME type validation to fail with `400 Unsupported media MIME type: image/jpeg`. After the TypeScript buffer fix, two additional pre-existing bugs were uncovered and fixed:

1. `minio>=7.2.0` requires `BytesIO` in `put_object`, not raw `bytes`
2. CV worker started via `Start-Process` without `.env` credentials → `AccessDenied` on MinIO writes

## Changes

### 1. `apps/api/src/media/media.service.ts` — Read from disk when buffer is undefined

**Problem**: `diskStorage` writes uploaded files to disk and sets `file.path`; `file.buffer` is `undefined`. The code used `file.buffer!` which was `undefined`, causing `buildMediaIngestionPlan()` to fail with `400 Bad Request`.

**Fix**: Added `readUploadedFileBuffer()` helper that reads from `file.buffer` when available, or from `file.path` when not. Also added temp file cleanup in the error path.

```typescript
private async readUploadedFileBuffer(file: UploadedFile): Promise<Buffer> {
  if (file.buffer) return file.buffer;
  if (file.path) {
    const fs = await import('fs/promises');
    return fs.readFile(file.path);
  }
  throw new InternalServerErrorException('No file content available for upload.');
}
```

Called before `buildMediaIngestionPlan()`:

```typescript
const buffer = await this.readUploadedFileBuffer(file);
plan = await buildMediaIngestionPlan({
  projectId,
  originalName: file.originalname,
  mimeType: file.mimetype,
  sizeBytes: file.size,
  buffer,
});
```

Error handler cleans up temp file:

```typescript
catch (err) {
  if (file.path) await this.safelyDeleteFile(file.path);
  // ...
}
```

### 2. `apps/cv-worker/src/storage.py` — BytesIO wrapper for minio>=7.2.0

**Problem**: `minio>=7.2.0` requires a readable IO object in `put_object` (e.g., `BytesIO`), not raw `bytes`. Passing `bytes` caused `'bytes' object has no attribute 'read'` at runtime.

**Fix**: Wrapped the `bytes` argument in `BytesIO()` before passing to `client.put_object()`.

```python
def write_object(storage_key: str, data: bytes, content_type: str) -> None:
    client = get_client()
    try:
        stream = BytesIO(data)
        client.put_object(
            bucket,
            storage_key,
            stream,
            len(data),
            content_type=content_type,
        )
```

### 3. `apps/cv-worker/src/main.py` — Load .env for Start-Process startup

**Problem**: When the CV worker is started via `Start-Process` (used by `start-dev.ps1`), the process does NOT inherit environment variables from the parent shell. This means `MINIO_ACCESS_KEY`, `MINIO_SECRET_KEY`, etc. are empty, causing `AccessDenied` on all MinIO operations.

**Fix**: Added `python-dotenv` to load `.env` at module import time, before any storage operations. Also made `get_client()` always re-read credentials from `os.environ` (not cached).

```python
from dotenv import load_dotenv

# From D:\Study\Project\Vision\apps\cv-worker\src\main.py up to root:
#   parents[0]=src, parents[1]=cv-worker, parents[2]=apps, parents[3]=Vision(root)
_env_path = Path(__file__).resolve().parents[3] / ".env"
load_dotenv(_env_path)
```

Also added `python-dotenv>=1.0.0` to `requirements.txt`.

### 4. `infra/docker-compose.yml` — MinIO bucket policy: private by default

**Problem**: The MinIO bucket `visionflow-artifacts` had no explicit policy set at startup, relying on default behavior which could allow unintended anonymous access.

**Fix**: Added `mc anonymous set none` to the `minio-init` command so the bucket is explicitly private from the moment the stack first starts. This ensures write operations always require authentication, not just after a manual `docker exec` fix.

```bash
mc anonymous set none local/visionflow-artifacts
```

### 5. `.env` — CV_WORKER_TIMEOUT_MS increased

**Problem**: The CV worker timeout was `3500ms`, but real thumbnail generation (reading from MinIO, Pillow processing, writing back to MinIO) takes ~6-10 seconds on first run.

**Fix**: Increased to `30000ms` (30 seconds).

```
CV_WORKER_TIMEOUT_MS=30000
```
