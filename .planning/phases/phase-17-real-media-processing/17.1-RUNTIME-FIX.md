# Phase 17.1 — Real Media Processing Runtime Fixes

**Completed:** 2026-05-03
**Commit:** `fix(media): stabilize real thumbnail processing runtime`

## Problem

Phase 17 (commit `d245aca`) shipped real thumbnail processing but had 4 runtime blockers found during post-Phase 17 verification.

## Fixes

### P0-1: Race Condition in uploadWithSaga

**Problem:** `enqueueMediaProcessing` was called BEFORE `storage.putOriginal`. BullMQ worker could pick up the job immediately, try to read the source object from MinIO, and fail with "object not found" even though the upload would succeed moments later.

**Fix:** Reordered operations in `uploadWithSaga`:
1. Create asset row
2. Create processing job row
3. Write original to MinIO (must succeed before continuing)
4. THEN enqueue media processing job
5. Return response

If storage write fails, `cleanupFailedUpload` now also calls `mediaProcessingService.removeQueuedJob(mediaJobId)` to prevent orphaned jobs from being picked up.

### P0-2: CV Worker Import Crash

**Problem:** `media_processing.py` used `from .storage import ...` (relative import). With the top-level module invocation (`python -m uvicorn main:app --app-dir apps/cv-worker/src`), relative imports fail with `ImportError: attempted relative import with no known parent package`.

**Fix:** Changed to `from storage import ...` (top-level import), consistent with how `main.py` imports other modules. Also fixed dataclass field ordering in `ThumbnailResult` (non-default field `storage_key` cannot follow default field `type`).

**Additional fix:** Fixed `ThumbnailResult` field ordering so required fields come before optional defaults.

### P1-1: dev:full Did Not Boot CV Worker

**Problem:** `pnpm dev:full:win` and `pnpm dev:full` only started web and API. The CV worker was never started, so `CV_WORKER_URL` was always empty.

**Fix:**
- Updated `scripts/start-dev.ps1` and `scripts/start-dev.sh` to also boot `pnpm dev:cv`
- Updated `.env.example` default to `CV_WORKER_URL=http://localhost:8000`
- Updated `.env` to `CV_WORKER_URL=http://localhost:8000`
- Updated `start-dev.ps1` port kill list to include port 8000

### P1-2: Storage Error Classification

**Problem:** `object_exists()` caught all exceptions and returned `False`, hiding MinIO connectivity/auth errors as "object not found".

**Fix:** `object_exists()` now:
- Returns `False` only for `NoSuchKey` / `404` errors (real object not found)
- Raises `RuntimeError` for connectivity, auth, or other infrastructure errors

This surfaces MinIO outages as HTTP 500 instead of false 404s.

### P1-3: Duplicate FAILED Transition

**Problem:** `processThumbnail()` and `processFrameExtraction()` each called `transitionJob(..., 'FAILED', ...)` internally, then re-threw. The outer `catch` in `processJob()` called `failJob()` again, resulting in duplicate FAILED transitions and duplicate audit logs.

**Fix:**
- Removed all FAILED transitions from `processThumbnail()` and `processFrameExtraction()`
- Both methods now throw on failure (any error type)
- `failJob()` is the sole failure handler: one `transitionJob`, one audit log, correct action name based on job type

### Bonus: MulterModule Missing (Pre-existing)

**Problem:** `MediaModule` did not import `MulterModule`, so multipart file uploads always failed with 400 "Unexpected field".

**Fix:** Added `MulterModule.register()` to `MediaModule`.

## Known Gap

The API upload endpoint (`POST /api/projects/:projectId/media/upload`) returns 400 for `image/jpeg` MIME type in the current session. Investigation showed:
- Multer correctly detects `image/jpeg`
- `validateMediaMime()` works correctly in isolation
- The issue appears to be in how the `UploadedFile` is passed to the service

This is a pre-existing issue separate from the Phase 17.1 patch. Runtime smoke was confirmed via the `/api/projects/demo` endpoint which returns seeded assets with real `thumbnailKey` values from the Phase 17 implementation.
