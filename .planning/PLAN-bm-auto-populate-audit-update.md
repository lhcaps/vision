## Plan Correction

The previous Phase 2 direction, "add hardcoded sample/default data to every BM",
is not production-grade and conflicts with the requirement DOCX. Replace it with:

1. Build a central `CasePayload -> BM form inputs` adapter layer.
2. Populate from live `render-payload`/case context first.
3. Preserve saved user edits as the highest-priority source.
4. Keep demo/sample buttons only as explicit demo actions, never as the default
   data path.
5. Add per-BM mapping metadata and coverage tests before touching 213 forms.

## Production Target Flow

1. Workspace fetches `render-payload` once.
2. `CasePayloadProvider` exposes normalized case, people, offenses,
   assignments, and agency data.
3. Each BM panel calls a shared adapter, for example
   `getBmInitialFormInputs(templateCode, casePayload, savedFormInputs)`.
4. Adapter order is: empty schema defaults -> case-derived defaults -> saved
   form inputs -> user edits.
5. UI shows a clear "Lấy từ vụ án" action for applying or re-applying
   case-derived fields.
6. Save persists only user-confirmed form inputs; render reads saved form inputs
   plus live payload, never stale hardcoded samples.

## Central Adapter Foundation (Shipped)

Pure helpers and one composition entry point now live under
`apps/web/src/lib/bm-auto-populate/`. Each helper has its own test
suite in `central-adapter.test.mjs`.

- `source-priority.ts`: 4-level priority stack
  `default -> case -> saved -> user`. Incoming non-empty wins; empty
  incoming preserves the lower-priority value, so Save/render never
  carries stale patched data.
- `case-field-resolver.ts`: 16 canonical case fields with safe
  fallbacks. Pure function, no React.
- `bm-field-map.ts`: per-BM `FieldTarget` registry. `BM-039` is
  registered; unknown BMs return an empty list so callers fall back to
  per-BM adapters until they migrate.
- `apply-case-payload-to-form.ts`: composition entry point. Clones the
  input form, applies the priority rule per target, returns the new
  form and the list of fields it changed.

`bm039-case-defaults` and `generic-case-defaults` were refactored to
use the central adapter while preserving their public API and existing
test contracts. New BMs can plug in by adding a `BM_FIELD_MAP` entry
and calling `applyCasePayloadToForm` from their own thin per-BM
adapter.

## Render Harness Fix

The previous merged `audit_renders/SUMMARY.md` claimed 213/213 OK while
`audit_renders/summary.json` only carried 113 OK codes. The two
artifacts were out of sync and the docx-sync audit (which drives the
readiness gate) was reading the JSON, so the readiness gate
incorrectly flagged a `render-json-incomplete` blocker.

Resolution shipped in this phase:

- `scripts/rebuild-bm-render-summary.mjs` rebuilds `summary.json` and
  `SUMMARY.md` from the 213 per-form `audit_renders/BM-XXX-result.json`
  artifacts, cross-checking the actual DOCX files on disk under
  `storage/generated/cases/VKS-2026-0001/docx/`. The output shape
  matches the previous `summary.json` (`startedAt`, `endedAt`,
  `apiHost`, `totalTemplates`, `results[]`, `errors[]`, `okCount`,
  `failedCount`, `withPlaceholders`).
- The legacy `summary.json` and `SUMMARY.md` are preserved under
  `audit_renders/.backup/` with a timestamped suffix. A rebuild
  metadata file is written to `audit_renders/.rebuild-info.json` and
  records: included count, excluded list, previous `summary.json`
  metrics, and a per-form disk verification record.
- Tool supports `--check` (dry run, exit 2 if incomplete), `--verbose`,
  `--render-dir`, `--docx-dir`, `--project-root`, and `-h`.
- Tests: `test/rebuild-bm-render-summary.test.mjs` (7/7 pass). Coverage:
  - 213/213 rebuild from on-disk artifacts
  - canonical summary.json shape (213 entries, 0 errors, startedAt from
    legacy, endedAt updated)
  - SUMMARY.md detail table has exactly 213 rows
  - legacy artifacts preserved under `.backup/`
  - `.rebuild-info.json` records the rebuild metadata
  - missing result file triggers `--check` exit code 2
  - missing DOCX file excludes the entry and surfaces it in `errors`

Readiness gate after the fix:

- `render-json-incomplete` blocker resolved.
- `renderSummaryJsonOkCount: 213`, `renderSummaryMdOkCount: 213`, 0
  json-vs-md mismatch codes.
- Four other blockers remain (`case-context-not-consumed`,
  `take-from-case-missing`, `generic-wrapper-panels`,
  `hardcoded-sample-risk`); these require the central adapter rollout
  and the per-BM field maps, not a render harness change.

## Dependency Direction

- Keep `docxtemplater` + `pizzip` for the current DOCX placeholder pipeline.
- Keep DOCX inspection dependency-free where possible by reading OOXML directly
  from zipped DOCX files.
- Use Playwright screenshot/snapshot checks for frontend preview regression when
  a browser/dev-server harness is available.
- For production DOCX-to-PDF conversion, prefer a service boundary around
  LibreOffice, such as Gotenberg or unoserver, instead of embedding ad-hoc
  conversion logic in the NestJS request path.
