# Phase D.0-D.1 Backend Foundation Audit

Date: 2026-06-19
Branch: `feat/phase-d-contract-runtime-foundation`

## Outcome

Phase D.0 and D.1 establish a stable backend foundation around the existing
NestJS application without changing the Prisma schema or contract corpus.

The backend now has:

- deterministic repository and storage path resolution;
- typed environment configuration with production safety checks;
- request correlation through `x-request-id`;
- a stable HTTP error envelope;
- typed application, configuration, and infrastructure errors;
- explicit liveness and readiness endpoints;
- a clean contract repository boundary for the DOCX contract corpus;
- schema validation, locked-contract precedence, caching, and diagnostics;
- a single import-upload storage boundary;
- authoritative root-level test orchestration;
- runtime smoke checks for health, readiness, web availability, and contract
  catalog integrity.

## Contract corpus capability

The application can read the contract corpus from:

`docs/audit/docx/contracts`

The runtime contract adapter:

1. resolves the contracts directory through `WorkspacePathsService`;
2. reads JSON contracts through `FileFormContractRepository`;
3. validates each document with Zod before exposing it;
4. excludes reference documents from the form catalog;
5. gives locked contracts precedence over draft contracts;
6. publishes catalog summaries and contract details through the forms API;
7. reports invalid files and missing locked pilot contracts through readiness.

Verified corpus state:

- total forms: 213;
- locked and runtime eligible: 3 (`BM-001`, `BM-002`, `BM-003`);
- draft and not runtime eligible: 210;
- invalid contracts: 0;
- reference documents exposed as forms: 0.

This means the backend has the contract-reading foundation required to create
forms. It does not mean all 213 forms are production-ready: only the three
locked pilot contracts are currently runtime eligible.

## Architecture boundary introduced

```text
HTTP controller
    |
application service
    |
FormContractRepository (port)
    |
FileFormContractRepository (filesystem adapter)
    |
docs/audit/docx/contracts
```

Shared infrastructure is supplied through the global `InfrastructureModule`:

- `AppConfigService`
- `WorkspacePathsService`

Business modules no longer read `process.env` or `process.cwd()` directly.
Bootstrap, seed, configuration providers, and the repository-root resolver
remain the intentional process-level boundaries.

## Operational endpoints

- `GET /api/v1/health` - process liveness.
- `GET /api/v1/ready` - contracts-root and locked-pilot readiness.
- `GET /api/v1/forms/catalog` - validated runtime catalog.
- `GET /api/v1/forms/catalog/:code` - validated contract detail.

Errors include:

- `statusCode`
- `code`
- `message`
- `requestId`
- `timestamp`
- `path`

The request ID is accepted from a safe incoming `x-request-id` or generated by
the backend and returned in both the response header and error body.

## Blockers removed

- development CORS did not reliably support loopback/LAN origins;
- public health and catalog routes were blocked by the global auth guard;
- frontend auth probing could emit unhandled network rejections;
- catalog loading mixed controller concerns, filesystem access, validation, and
  precedence rules in one service;
- repository paths depended on the process launch directory;
- import upload storage bypassed the shared path boundary;
- auth cookie/session settings were evaluated directly from environment
  variables in feature code;
- LibreOffice path discovery bypassed typed configuration;
- the existing root test command selected incompatible test runners;
- BM-039 auto-population failed to replace a known default placeholder.

## Remaining architecture debt

The foundation is complete, but the backend is not yet fully decomposed.

The largest remaining hotspot is
`document-renderer.service.ts` at approximately 33,704 lines. It still combines
template-specific mapping, rendering policy, formatting, and orchestration.
The API module tree also contains approximately 2,265 occurrences of `any`.

Recommended Phase D.2:

1. place new rendering work behind a `DocumentRenderPort`;
2. extract template-family strategies incrementally, beginning with locked
   contracts;
3. separate canonical data mapping from DOCX formatting;
4. add golden-file rendering tests before moving each strategy;
5. reduce `any` at module boundaries with DTO and canonical-context types;
6. leave the legacy renderer as a compatibility adapter until parity is proven.

Do not rewrite the renderer wholesale. Move one verified template family at a
time behind stable interfaces.
